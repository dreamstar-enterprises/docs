    Using the mix and elixir commands

    Guides • Erlang & Elixir • 04/12/2021

    Introduction

    So far, the command iex -S mix has been used to start the system. It’s also possible to start the system with mix run --no-halt. This command starts the BEAM instance and then starts the OTP application together with its dependencies. The --no-halt option instructs mix to keep the BEAM instance running forever.

    Elixir 
    `$ mix run --no-halt  # Starts the system without the iex shel

Starting to-do cache

    Compared to iex -S mix, the important difference is that mix run doesn’t start the interactive shell.

    A slightly more elaborate option is to use the elixir command:

    Elixir 
    `$ elixir -S mix run --no-halt

Starting to-do cache

    This approach requires a bit more typing, but it allows you to run the system in the background.

    Detached Mode

    By using the --detached flag, you can start the system in detached mode. The OS process will be detached from the terminal, and there will be no console output (it’s redirected to /dev/null). When starting a detached system, it’s also useful to turn the BEAM instance into a node, so you can later interact with it and terminate it when needed:

    Elixir 
    `$ elixir --detached --sname todo_system@localhost -S mix run --no-halt

    This starts the BEAM instance in background.

    You can check that it’s running by looking at which BEAM nodes exist on the system:

    Elixir 
    `$ epmd -names

epmd: up and running on port 4369 with data:
name todo_system at port 51028   # The node is running

    At this point, the system is running and you can use it — for example, by issuing an HTTP request to manipulate to-do lists.

    Remote Shell

    You can connect to a running BEAM instance and interact with it. It’s possible to establish a remote shell — something like a terminal shell session to the running BEAM instance. In particular, with the the --remsh option you can start another node and use it as a shell to the todo_system node:

    Elixir 
    `$ iex --sname debugger@localhost --remsh todo_system@localhost --hidden

iex(todo_system@localhost)1>  # Shell is running on the todo_system node

    In this example, debugger node is started, but the shell is running in the context of todo_system. Whatever function you call will be invoked on todo_system. This is extremely useful, because you can now interact with the running system. BEAM provides all kinds of nice services that allow you to query the system and individual processes.

    The debugger node is started as hidden. This means the debugger node won’t appear in the results of Node.list (or Node.list([:this, :visible])) on todo_system, so it won’t be considered part of the cluster.

    Stopping Shell

    To stop the running system, you can use the System.stop function HexDocs - System Stop, which takes down the system in a graceful manner. It shuts down all running applications and then terminates the BEAM instance.

    Elixir 
    iex(todo_system@localhost)1> System.stop()

    The remote shell session is left hanging, and an attempt to run any other command will result in an error:

    Elixir 
    iex(todo_system@localhost)2>
*** ERROR: Shell process terminated! (^G to start new job) ***

    At this point, you can close the shell and verify the running BEAM nodes:

    Elixir 
    `$ epmd -names
epmd: up and running on port 4369 with data:

    If you want to stop a node programmatically, you can rely on distributed features. Here’s a quick example:

    Elixir 
    if Node.connect(:todo_system@localhost) == true do
   :rpc.call(:todo_system@localhost, System, :stop, [])  # Invokes System.stop on a remote node
   IO.puts "Node terminated."
else
   IO.puts "Can't connect to a remote node."  
end

    Here, you connect to a remote node and then rely on :rpc.call/4 to invoke System.stop there.

    You can store the code in the stop_node.exs file (the .exs extension is frequently used for Elixir-based scripts). Then you can run the script from the command line:

    Elixir 
    `$ elixir --sname terminator@localhost stop_node.exs

    Running a script starts a separate BEAM instance and interprets the code in that instance. After the script code is executed, the host instance is terminated. Because the script instance needs to connect to a remote node (the one you want to terminate), you need to give it a name to turn the BEAM instance into a proper node.
