    Tail Function Calls

    Guides • Erlang & Elixir • 04/12/2021

    Introduction

    If the last thing a function does is call another function (or itself), it is a tail call:

    Elixir 
    def original_fun(...) do
  ...
  another_fun(...)            
end

    Elixir treats tail calls in a specific manner by performing a tail-call optimization.

    In this case, calling a function doesn’t result in the usual stack push, but instead something like a goto or a jump statement happens. Additional stack space isn’t allocated before calling the function, which in turn means the tail function call consumes no additional memory.

    In the previous snippet, the last thing done in original_fun is the call of another_fun. The final result of original_fun is the result of another_fun. This is why the compiler can safely perform the operation by jumping to the beginning of another_fun without doing additional memory allocation. When another_fun finishes, you return to whatever place in original_fun that it was called from.

    Tail calls are especially useful in recursive functions. A tail-recursive function — that is, a function that calls itself at the very end — can run virtually forever without consuming additional memory.

    The following function is the Elixir equivalent of an endless loop:

    Elixir 
    def loop_forever(...) do
  ...
  loop_forever(...)
end

    Because tail recursion doesn’t consume additional memory, it’s an appropriate solution for arbitrarily large iterations. There is a downside, though. Whereas classical (non-tail) recursion has a more declarative feel to it, tail recursion tends to looks more procedural.

    The following converts the ListHelper.sum/1 function to the tail-recursive version.

    Elixir 
    defmodule ListHelper do
  def sum(list) do
    do_sum(0, list)
  end

  defp do_sum(current_sum, []) do
    current_sum
  end

  defp do_sum(current_sum, [head | tail]) do
    new_sum = head + current_sum
    do_sum(new_sum, tail)
  end
end

    There are two functions. The exported function sum/1 is called by the module clients, and on the surface it works like before.

    The recursion takes place in the private do_sum/2 function, which is implemented as tail-recursive. It’s a two-clause function. Here is the second clause in isolation:

    Elixir 
    defp do_sum(current_sum, [head | tail]) do
  new_sum = head + current_sum                   
  do_sum(new_sum, tail)                          
end

    This clause expects two arguments: the non-empty list to operate on, and the sum calculated so far (current_sum). It then calculates the new sum and calls itself recursively with the remainder of the list and the new sum. Because the call happens at the very end, the function is tail-recursive, and the call consumes no additional memory.

    The variable new_sum is introduced here just to make things more obvious. You could also inline the computation:

    Elixir 
    defp do_sum(current_sum, [head | tail]) do
  do_sum(head + current_sum, tail)
end

    This function is still tail-recursive because it calls itself at the very end.

    For the first clause:

    Elixir 
    defp do_sum(current_sum, []) do
  current_sum
end

    This clause stops the recursion. It matches on an empty list, which is the last step of the iteration. When you get here, there’s nothing else to sum, so it returns the accumulated result.

    For the function sum/1:

    Elixir 
    def sum(list) do
  do_sum(0, list)
end

    This function initializes the value of the current_sum parameter that’s passed recursively in do_sum.

    The tail recursion is a direct equivalent of a classical loop in other languages.

    The parameter current_sum is a classical accumulator: the result is incrementally added to it, in each iteration step. The do_sum/2 function implements the iteration step and passes the accumulator from one step to the next. The first clause of do_sum/2 defines the ending point of the iteration and returns the accumulator value.

    Testing the function from the shell, you should get:

    Elixir 
    iex(1)> ListHelper.sum([1, 2, 3])
6

iex(2)> ListHelper.sum([])
0

    Recognising Tail Calls

    Tail calls can take different shapes - e.g. they can also happen in a conditional expression:

    Elixir 
    def fun(...) do
  ...
  if something do
    ...
    another_fun(...)     
  end
end

    The call to another_fun is a tail call because it’s a last thing the function does. The same rule holds for unless, cond, case, and with expressions.

    The following code isn’t a tail call:

    Elixir 
    def fun(...) do
  1 + another_fun(...)    
end

    This is because the call to another_fun isn’t the last thing done in the fun function. After another_fun finishes, its result is incremented by 1 to compute the final result of fun.

    Examples

    Here are some more examples:

    • A list_len/1 function that calculates the length of a list
    • A range/2 function that takes two integers, from and to, and returns a list of all numbers in the given range
    • A positive/1 function that takes a list and returns another list that contains only the positive numbers from the input list
    Elixir 
    defmodule Loop do
  def list_len([]), do: 0

  def list_len([_ | tail]) do
    1 + list_len(tail)
  end

  def range(from, to) when from > to do
    []
  end

  def range(from, to) do
    [from | range(from + 1, to)]
  end

  def positive([]), do: []

  def positive([head | tail]) when head > 0 do
    [head | positive(tail)]
  end

  def positive([_ | tail]) do
    positive(tail)
  end
end

    These are the tail-call equivalent versions:

    Elixir 
    defmodule Loop do
  def list_len(list), do: calc_list_len(list, 0)

  defp calc_list_len([], len), do: len

  defp calc_list_len([_ | tail], len) do
    calc_list_len(tail, len + 1)
  end


  def range(from, to) do
    make_range(from, to, [])
  end

  defp make_range(from, to, result) when from > to do
    result
  end

  defp make_range(from, to, result) do
    make_range(from, to - 1, [to | result])
  end


  def positive(list) do
    filter_positive(list, [])
  end

  defp filter_positive([], result) do
    Enum.reverse(result)
  end

  defp filter_positive([head | tail], result) when head > 0 do
    filter_positive(tail, [head | result])
  end

  defp filter_positive([_ | tail], result) do
    filter_positive(tail, result)
  end
end

    Recursion is the basic looping technique, and no loop can be done without it.

    Many typical tasks also can be performed by using higher-order functions.
